### 二、TokenAudit 完整架构设计（前后端+多Agent）
#### （一）整体架构流程图
1.  前端层（Vue3 + Element-Plus）：用户录入待审计Token、中转平台、宣称模型等参数，发起审计请求，展示审计报告、Token管理列表、历史审计记录，支持报告导出（PDF/Excel）。
2.  后端层（SpringBoot）：接收前端请求，封装DeepSeek API调用接口、待审计Token调用接口，实现多Agent调度、数据持久化（存储审计记录、Token信息、报告数据），提供权限控制（避免审计接口滥用）。
3.  审计核心层（Python + 多Agent + LangGraph）：由5个专用审计Agent组成，通过LangGraph实现任务调度，依托DeepSeek API Key调用DeepSeek模型完成特征比对、逻辑判定，输出审计原始数据。
4.  数据层：存储审计基础信息、分维度测试记录、综合审计报告、Token生命周期数据，支持历史查询、数据统计。

#### （二）多Agent分工（基于Python + LangGraph实现，依托DeepSeek基座）
1.  调度Agent（Orchestrator Agent）：核心调度节点，接收后端请求，分配任务给其他审计Agent，汇总各Agent审计结果，提交给DeepSeek模型进行综合判定，确保审计流程有序执行（不可遗漏任何审计项）。
2.  有效性审计Agent（ValidityAgent）：调用待审计Token对应的中转API，执行3次不同指令调用，记录响应状态码、响应时间，依托DeepSeek模型判断Token有效性、过期状态、封禁情况，输出原始测试数据。
3.  权限审计Agent（PermissionAgent）：检测待审计Token的调用权限范围，测试“宣称模型调用”“非宣称模型调用”“匿名调用”三种场景，结合DeepSeek模型的逻辑推理能力，判定权限是否正常、是否存在越界/不足。
4.  掺水审计Agent（WateringAgent）：调用待审计Token获取模型响应，提取响应特征（格式、推理能力、知识截止日期等），与DeepSeek模型中存储的“宣称模型原生特征库”进行比对，判定是否存在掺水、降级、套皮行为。
5.  合规性与稳定性审计Agent（ComplianceStabilityAgent）：检测Token泄露风险、复用性、调用频率，执行5次同一指令调用测试稳定性，依托DeepSeek模型分析测试数据，输出合规性、稳定性判定结果。

### 三、核心审计任务（必须全部执行，不可遗漏，依托架构落地）
1.  Token有效性审计（由有效性审计Agent执行，DeepSeek辅助判定）：
    - 测试逻辑：通过Python脚本调用待审计Token对应的中转API，执行3次不同指令（基础指令、代码指令、长文本指令），记录每次调用的响应状态码、响应时间、响应内容。
    - 辅助判定：将测试记录提交给DeepSeek模型，由DeepSeek结合状态码、响应信息，判定Token是否有效、是否过期、是否被封禁。
2.  Token权限审计（由权限审计Agent执行，DeepSeek辅助判定）：
    - 测试逻辑：分别调用“宣称模型”“非宣称模型”，尝试匿名调用（不携带Token），记录调用结果。
    - 辅助判定：DeepSeek模型根据测试结果，判定权限是否与宣称一致，是否存在越界、不足、匿名权限异常。
3.  Token掺水检测关联审计（由掺水审计Agent执行，DeepSeek核心判定）：
    - 测试逻辑：调用待审计Token获取模型响应，提取响应特征（格式、推理能力、知识截止日期、输出风格），与DeepSeek模型中存储的“宣称模型原生特征库”比对。
    - 核心判定：DeepSeek模型分析特征差异，判定是否存在“高价Token对应低价模型”“掺水调用”“混合模型调用”等异常。
4.  Token使用合规性审计（由合规性与稳定性审计Agent执行，DeepSeek辅助判定）：
    - 测试逻辑：检测API响应中是否包含待审计Token本身，测试多设备复用Token是否成功，短时间内高频调用（10次/分钟）观察是否异常。
    - 辅助判定：DeepSeek模型根据测试结果，判定Token使用是否合规，是否存在泄露、非法复用风险。
5.  Token稳定性审计（由合规性与稳定性审计Agent执行，DeepSeek辅助判定）：
    - 测试逻辑：连续5次调用同一指令（如“请输出当前模型名称”），记录每次响应时间、响应结果、是否报错。
    - 辅助判定：DeepSeek模型分析响应时间波动、输出一致性、报错频率，判定Token对应的API服务是否稳定。

### 四、审计判定标准（结合DeepSeek判定结果，精准可落地）
#### （一）Token有效性判定
- 有效：3次调用均成功（状态码200），响应时间正常（≤3s），无过期、封禁提示，DeepSeek判定为有效；
- 无效：调用失败（状态码401/403/500等），提示Token过期、无效或封禁，DeepSeek判定为无效；
- 异常：部分调用成功，部分失败，响应时间波动极大（＞5s），DeepSeek判定为异常。

#### （二）Token权限判定
- 权限正常：调用权限与宣称一致，无越界、无不足，DeepSeek判定为权限正常；
- 权限异常：权限越界（可调用未宣称模型）、权限不足（无法调用宣称模型），DeepSeek判定为权限异常；
- 匿名权限异常：无需Token即可调用，或Token无校验机制，DeepSeek判定为匿名权限异常。

#### （三）Token掺水关联判定
- 无掺水：Token调用的实际模型与宣称模型完全匹配，响应特征、能力一致，DeepSeek判定为无掺水；
- 疑似掺水：部分调用匹配宣称模型，部分调用为低等级模型，响应特征不稳定，DeepSeek判定为疑似掺水；
- 确认掺水：所有调用均为低等级模型，与宣称模型差异极大，存在明显降级、套皮，DeepSeek判定为确认掺水。

#### （四）合规性判定
- 合规：无Token泄露，无非法复用，调用频率正常，DeepSeek判定为合规；
- 不合规：响应中泄露Token、Token可被多设备复用、调用频率异常（疑似共享），DeepSeek判定为不合规。

#### （五）稳定性判定
- 稳定：5次调用均成功，响应时间波动≤1s，输出一致，无报错，DeepSeek判定为稳定；
- 不稳定：存在1-2次报错/超时，响应时间波动＞2s，输出存在轻微差异，DeepSeek判定为不稳定；
- 极不稳定：3次及以上报错/超时，响应时间波动＞3s，输出差异极大，DeepSeek判定为极不稳定。
